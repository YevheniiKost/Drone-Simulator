# Git: гілки, коміти, робочий процес

Цей документ задає правила для повідомлень комітів, назв гілок і загальної роботи з Git у репозиторії. Дотримуйся його, коли пропонуєш коміти, назви гілок або кроки роботи з Git.

## Повідомлення комітів

- Формат заголовка: `тип(область): короткий опис` за духом [Conventional Commits](https://www.conventionalcommits.org/).
- Поширені типи: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`.
- `область` — необов’язково; коротко вказує модуль або шар (наприклад `ui`, `physics`, `config`, `presentation`).
- Заголовок — імператив, як у повідомленнях ядра Git (наприклад *Add*, *Fix*, *Remove*), не минулий час.
- Без крапки в кінці заголовка; бажано в межах ~72 символів для читабельного `git log`.
- Один коміт — одна логічна зміна; не змішуй неспоріднені правки в одному коміті, якщо їх можна розділити.
- Якщо *чому* зміни неочевидно з дифу, додай тіло: порожній рядок після заголовка, далі пояснення.

Мова повідомлень комітів: узгоджуй з командою; за замовчуванням для цього репозиторію — **англійська** в заголовку (зручно для інструментів і чужих рев’ю), якщо команда не вирішила інакше.

## Гілки

- Префікси за призначенням: `feature/`, `fix/`, `chore/`, `release/` (або спрощений набір, який прийняла команда).
- Ім’я в **kebab-case**, коротко й описово: `feature/hud-input-panel`, `fix/motor-torque-clamp`.
- Уникай дуже довгих назв; деталі — в issue або описі PR.

## Перед комітом

- Переглянь `git status` і `git diff`; не коміть випадкові файли (кеш IDE, локальні налаштування, тимчасові ассети).
- Тримай `.gitignore` актуальним для Unity та середовища розробки.
- Не коміть секрети (ключі API, паролі, приватні сертифікати).

## Pull request і рев’ю

- Опис PR пояснює *що* змінено і за потреби *як перевірити*.
- Велика зміна — краще кілька осмислених комітів, ніж один нечитабельний «все в одному».

## Злиття з основною гілкою

- Політику **merge vs rebase** дотримуй тієї, яку прийняла команда для цього репозиторію; не переписуй публічну історію без узгодження.

## Приклади заголовків

- `feat(ui): add HUD window shell`
- `fix(physics): clamp motor torque to config max`
- `chore: update packages lock file`
